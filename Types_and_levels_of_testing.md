## Категории тестирования

### По стадии проведения

- Unit-тестирование
- Интеграционное тестирование
- Системное тестирование
- Альфа-тестирование
- Бета-тестирование
- Smoke-тестирование
- Регрессионное тестирование
- Приёмочное (UAT) тестирование

### По знанию устройства системы

- **Black-box**: Без знания внутреннего кода (тестировщик видит только интерфейс)
- **White-box**: Полное знание кода и архитектуры (только разработчики)
- **Grey-box**: Частичное знание (доступ к БД, логам)

### По направленности сценария

- **Функциональное**: Проверка функций и требований
- **Нефункциональное**:
  - Performance (производительность)
  - Security (безопасность)
  - Usability (удобство)
  - Compatibility (совместимость)
  - Reliability (надежность)
  - Accessibility (доступность)

### По выполнению кода

- **Статическое**: Анализ без выполнения кода (code review, анализ макетов)
- **Динамическое**: Выполнение кода и проверка поведения

## Уровни тестирования

### 1. Модульное (Unit) тестирование

**Что проверяем**: Отдельные функции или компоненты на этапе разработки

**Кто проводит**: Разработчики

**Артефакты**:

- Исходный код модуля
- Юнит-тесты
- Логика функций

**Типичные дефекты**:

- Ошибка арифметического расчета
- Неправильная валидация пароля
- Исключение при пустом входном параметре

**Пример**: Проверка функции расчета комиссии за перевод

---

### 2. Интеграционное (Integration) тестирование

**Что проверяем**: Взаимодействие нескольких модулей между собой

**Кто проводит**: Тестировщики и разработчики

**Артефакты**:

- Связанные модули
- Заглушки внешних сервисов
- API спецификации

**Типичные дефекты**:

- Неверная передача данных между модулями
- Сбой при одновременной обработке операций
- Ошибка формата данных при обмене с внешним сервисом

**Пример**: Проверка взаимодействия модуля переводов с модулем уведомлений

---

### 3. Системное (System) тестирование

**Что проверяем**: Вся система целиком на этапе внутреннего тестирования

**Кто проводит**: Тестировщики

**Артефакты**:

- Полная сборка приложения
- Тестовые сценарии пользователя
- Базы данных / тестовые окружения

**Типичные дефекты**:

- Краш приложения при входе через определённое устройство
- Некорректное отображение интерфейса на маленьких экранах
- Потеря данных при параллельных операциях

**Пример**: Проверка полного процесса платежа (вход → выбор товара → оплата)

---

### 4. Приёмочное (UAT) тестирование

**Что проверяем**: Соответствие бизнес-требованиям, готовность к выпуску

**Кто проводит**: Заказчик, QA, бизнес-аналитики

**Артефакты**:

- ТЗ/требования
- Полная сборка приложения
- Ручные сценарии проверки

**Типичные дефекты**:

- Функция не соответствует требованиям лимитов
- Не учитывается локализация валюты
- Отсутствует подтверждение важной операции

**Пример**: Проверка соответствия требованиям по переводам (лимиты, комиссии, отчеты)

## Анализ требований

### Критерии качества требования

#### 1. Полнота ❌❌❌

Требование должно содержать все необходимые детали, без которых невозможно создать или протестировать.

**Пример проблемы**:

- "Поле ввода названия магазина принимает любые символы" ❌
- Что происходит при вводе пробелов, спецсимволов, очень длинных строк (максимум символов?)

**Исправленная версия**:

- "Поле принимает буквы, цифры, ограничено 50 символами, спецсимволы кроме '@' и '#'" ✅

#### 2. Однозначность ✅

Требование не должно допускать двоякой интерпретации.

**Пример проблемы**:

- "При повторном открытии приложения восстанавливается сохранённое состояние" ❌
- Какой интервал повторов? Сразу ли после закрытия или через час?

#### 3. Атомарность ✅

Требование должно проверять ОДНУ функцию, а не несколько.

**Пример проблемы**:

- "На карте отображается выбранный флаг маршрута со стрелкой определённого цвета и вида" ❌
- Разбить на 3 требования:
  1. Флаг выбирается при клике
  2. Стрелка отображается с правильным видом
  3. Стрелка имеет правильный цвет

**Пример проблемы**:

- "Пользователь может построить маршрут и отменить его" ❌
- Разбить на 2:
  1. Пользователь может построить маршрут
  2. Пользователь может отменить маршрут

#### 4. Непересекаемость ✅

Требование не должно конфликтовать с другими требованиями той же категории.

#### 5. Проверяемость ✅

Требование должно быть проверяемым, т.е. есть четкое условие успеха/неудачи.

## Чек-листы Black-box тестирования

### Приложение "Найди и иди" - Поле ввода названия магазина

```
☐ Поле принимает буквы (a-z, A-Z, а-я)
☐ Поле принимает цифры (0-9)
☐ Поле принимает спецсимволы (!@#$%)
☐ Поле принимает пробелы
☐ Плейсхолдер отображается в пустом поле
☐ Плейсхолдер исчезает при вводе первого символа
☐ Плейсхолдер возвращается после очистки поля
☐ История поиска сохраняется и доступна при клике на поле
☐ При вводе 1 символа кнопка "Поиск" остаётся неактивной
☐ При вводе 2+ символов кнопка "Поиск" становится активной
☐ Поле принимает очень длинные строки (100+ символов)
☐ Поле принимает только пробелы (требует уточнения ТЗ)
```

### Кнопка "Поиск"

```
☐ Кнопка неактивна при незаполненных обоих полях
☐ Кнопка становится активной при заполнении поля названия (2+ символа)
☐ Кнопка становится активной при выборе типа магазина
☐ Кнопка остаётся активной при заполнении обоих полей
☐ Поиск выполняется при клике на активную кнопку
☐ Результаты отображаются флажками на карте
☐ При поиске без результатов флажков нет на карте
☐ Карта скроллится (вверх, вниз, влево, вправо)
☐ Карта зуммируется при pinch-жесте
```

### Общие требования

```
☐ Незаконченный маршрут восстанавливается при повторном открытии приложения
☐ История поиска восстанавливается при повторном открытии приложения
☐ Геопозиция работает при нормальном уровне батареи
☐ Геопозиция отключается при критическом уровне батареи
☐ Приложение не падает при быстром открытии/закрытии
☐ Приложение не падает при смене ориентации экрана
```

## Чек-листы Grey-box тестирования

### Предусловие

Есть доступ к базам данных, где хранится информация о магазинах.

### Поле ввода названия магазина

```
☐ Поиск возвращает магазины, существующие в БД
☐ Поиск не возвращает магазины, отсутствующие в БД
☐ Поиск корректно обрабатывает различные варианты ввода названия
☐ Поиск возвращает все магазины с одинаковым названием из БД
☐ Поиск не возвращает архивные / неактивные магазины
```

### Кнопка "Построить маршрут"

```
☐ Маршрут строится только для магазинов с валидными координатами в БД
☐ Магазины без координат не позволяют построить маршрут
☐ Некорректные координаты не вызывают падение приложения
```

### Область карты

```
☐ Все флажки на карте соответствуют записям в БД
☐ Координаты флажков совпадают с координатами в БД
☐ Магазины с одинаковыми координатами корректно отображаются
☐ Магазины, находящиеся в архиве БД, не отображаются на карте
```

### Общие требования

```
☐ История поиска соответствует реальным данным из БД
☐ Удаленный из БД магазин не появляется в истории
☐ Приложение корректно работает при большом количестве записей
```

## Лучшие практики

1. **Комбинируйте подходы**: Black-box дает пользовательскую перспективу, grey-box выявляет интеграционные проблемы
2. **Анализируйте требования перед тестированием**: Найденные пробелы могут сэкономить дни работы
3. **Используйте чек-листы для быстрого тестирования**: регрессия, smoke tests
4. **Документируйте найденные противоречия**: Помогает улучшить процесс в будущем
