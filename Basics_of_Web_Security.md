## Матрица рисков безопасности

### Пример: Безопасный чат врачей и пациентов

| № | Принцип                       | Сценарий/риск                      | Причина                                                 | Тип риска |
| -- | ------------------------------------ | ---------------------------------------------- | -------------------------------------------------------------- | ----------------- |
| 1  | Конфиденциальность | Перехват сообщений (MITM)     | Отсутствие TLS/SSL                                   | Высокий    |
| 2  | Конфиденциальность | Доступ через SQL-инъекцию   | Непроверенные параметры                  | Высокий    |
| 3  | Конфиденциальность | Утечка через логирование | Логи хранят тексты без маскировки | Средний    |
| 4  | Конфиденциальность | XSS в сообщениях                    | Отсутствие экранирования                | Средний    |
| 5  | Конфиденциальность | Кража учетных данных         | Отсутствие 2FA                                       | Высокий    |
| 6  | Целостность               | Модификация сообщений      | Отсутствие подписи/хеша                   | Высокий    |
| 7  | Целостность               | Вредоносные файлы              | Отсутствие проверки файлов             | Высокий    |
| 8  | Целостность               | Изменение истории чата     | Непрозрачный лог действий               | Средний    |
| 9  | Доступность               | DDoS-атака на сервер              | Отсутствие защиты                              | Высокий    |
| 10 | Доступность               | Потеря доступа (auth сбой)    | Единая точка отказа                           | Средний    |

### Топ-3 критических рисков

1. **Перехват сообщений (MITM)**

   - Компрометация всей коммуникации
   - Раскрытие конфиденциальной информации
2. **SQL-инъекция в БД**

   - Получение, изменение, удаление всех данных
   - Доступ к чатам и личной информации
3. **DDoS-атака на сервер**

   - Недоступность сервиса в критический момент
   - Может привести к вреду здоровью пациента

### Рекомендации

- ✅ Обязательное использование TLS 1.2/1.3
- ✅ HTTP Strict Transport Security (HSTS)
- ✅ Pinning сертификатов
- ✅ Параметризованные запросы (защита от SQL-инъекций)
- ✅ Валидация и санитайзинг входных данных
- ✅ Rate limiting и защита от DDoS

## Кодирование vs Хеширование vs Шифрование

| Параметр                 | Кодирование                                    | Хеширование                                     | Шифрование                                                      |
| -------------------------------- | --------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------- |
| **Назначение**   | Представление в другом формате | Создание уникального отпечатка | Защита от несанкционированного доступа |
| **Обратимость** | Да (двусторонний)                           | Нет (одностороннее)                        | Да (с ключом)                                                    |
| **Ключ**               | Не требуется                                   | Не требуется                                    | Требуется                                                        |
| **Примеры**         | Base64, URL-кодирование                        | MD5, SHA, bcrypt                                           | AES, RSA, DES                                                             |

### Примеры использования

**Кодирование**: Base64 для передачи бинарных данных в JSON API

```
http://example.com/api?data=SGVsbG8gV29ybGQ=
```

**Хеширование**: Хранение пароля в БД

```
Пароль: "password123"
Хеш: $2b$12$... (bcrypt)
При входе: хешируем введённый пароль и сравниваем хеши
```

**Шифрование**: Защита платежных данных

```
Номер карты: 4111 1111 1111 1111
Зашифровано: AES-256 перед передачей на сервер платежей
```

## Симметричное vs Асимметричное шифрование

### Аналогия

**Симметричное** = Обычный замок на двери

- Один ключ открывает и закрывает замок
- Быстро, но нужно как-то передать ключ другим людям

**Асимметричное** = Система почтовых ящиков

- Публичный ключ (адрес ящика - все его знают)
- Приватный ключ (только хозяин имеет)
- Любой может положить письмо (зашифровать)
- Только хозяин может открыть (расшифровать)

### Сравнительная таблица

| Параметр                 | Симметричное                                 | Асимметричное                                          |
| -------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------- |
| **Определение** | Один ключ для шифр/расшифр         | Два ключа: публичный и приватный         |
| **Плюсы**             | Быстро, малый размер ключей       | Безопасная передача, аутентификация |
| **Минусы**           | Проблема распределения ключей | Медленно, большие ключи                         |
| **Примеры**         | AES, 3DES, ChaCha20                                      | RSA, ECDSA, Diffie-Hellman                                          |

### Где используется

**Симметричное** (TLS): AES-GCM, ChaCha20

- Быстро для передачи большых объемов данных

**Асимметричное** (TLS Handshake): RSA, ECDHE

- Для безопасного обмена симметричными ключами

## Чёрный и белый списки алгоритмов

### ❌ НЕБЕЗОПАСНЫЕ алгоритмы (Чёрный список)

#### Симметричное шифрование

**DES (Data Encryption Standard)**

- ❌ Слишком короткий ключ (56 бит)
- ❌ Брутфорс за часы на современном ПК
- ❌ Предсказуемые паттерны

**RC4 (Rivest Cipher 4)**

- ❌ Ошибки в инициализации
- ❌ Ненулевые вероятности в потоке ключей
- ❌ Уязвимость для атак на SSL/TLS

**ECB (Electronic Codebook mode)**

- ❌ Одинаковые блоки открытого текста → одинаковые блоки шифра
- ❌ Видны паттерны (например, силуэт на зашифрованном изображении)

#### Хеширование

**MD5 (Message Digest 5)**

- ❌ Найдены коллизии (два разных файла с одинаковым хешем)
- ❌ Использовалась для подделки SSL сертификатов
- ❌ Быстрое хеширование позволяет ускоренный брутфорс

**SHA-1 (Secure Hash Algorithm 1)**

- ❌ Практические коллизии найдены в 2017 году
- ❌ Удалена из SSL сертификатов (браузеры больше не доверяют)
- ❌ Меньше защиты против атак "дней рождения"

**CRC32**

- ❌ Не криптографическая функция (предназначена для проверки ошибок)
- ❌ Легко перебирается (2^32 = 4 млрд комбинаций)
- ❌ Линейность позволяет манипулировать данными

---

### ✅ БЕЗОПАСНЫЕ алгоритмы (Белый список)

#### Симметричное шифрование

**AES (Advanced Encryption Standard) - Rijndael**

- ✅ Утвержден правительством США для "TOP SECRET"
- ✅ Ключи 128, 192, 256 бит (невозможно перебрать)
- ✅ 20+ лет без критических уязвимостей
- ✅ Используется везде (банки, гос. учреждения, военные)

**ChaCha20**

- ✅ Быстрее на устройствах без AES-NI (мобильные, старые ПК)
- ✅ Стойкая математика против differential & linear cryptanalysis
- ✅ Принята как стандарт в TLS 1.3 и WireGuard VPN

#### Хеширование

**SHA-256 (SHA-2 семейство)**

- ✅ 256-бит выход (защита на уровне 2^128 попыток)
- ✅ Коллизии не найдены (20+ лет исследований)
- ✅ Используется в Bitcoin
- ✅ Стандарт NIST и ISO

**bcrypt**

- ✅ Специально для паролей (медленное хеширование)
- ✅ Встроена "соль" (защита от rainbow tables)
- ✅ Адаптивный cost factor (можно увеличивать при росте ПК)
- ✅ Используется в production везде (Django, Ruby, PHP)

## Общественный компьютер - забыл разлогиниться

### Как проверить

1. **Проверка восстановления сессии**

   - Закрыть браузер без явного logout
   - Открыть браузер заново
   - Проверить, остались ли куки → возможен ли доступ в кабинет?
2. **Проверка куков в DevTools**

   - F12 → Application → Cookies
   - Посмотреть флаги: Secure, HttpOnly, SameSite, Expires
3. **Перехват через Burp Suite**

   - Войти в аккаунт
   - Закрыть браузер
   - Посмотреть, какие куки передавались
4. **Попытка доступа со старой сессией**

   - Сохранить cookie после входа
   - На другом браузере подменить cookie
   - Можно ли зайти в кабинет другого пользователя?

### Что должно происходить

| Мера защиты                                          | Как проверить                                            | Ожидаемое                                                |
| -------------------------------------------------------------- | -------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **Auto logout при закрытии браузера** | Закрыть браузер, открыть → нужен ввод | ✅ Требуется повторный вход                 |
| **HttpOnly флаг**                                    | DevTools → Cookies                                                  | ✅ Cookie отмечен как HttpOnly                          |
| **Secure флаг**                                      | DevTools → Cookies                                                  | ✅ Cookie только по HTTPS                                 |
| **SameSite (Strict/Lax)**                                | DevTools → Cookies                                                  | ✅ SameSite=Strict или Lax                                     |
| **Session Timeout**                                      | Оставить открытым на 30 минут                 | ✅ Сессия истекает, требуется вход     |
| **Проверка User-Agent**                          | Войти на Windows, cookie на Mac                             | ✅ Обнаружение смены, переauthentication      |
| **Проверка IP**                                  | Войти с IP A, cookie с IP B                                   | ✅ При смене IP требуется переauthentication |
| **CSRF токены**                                    | Отправить форму без CSRF-токена               | ✅ Отказ 403 Forbidden                                       |
| **Явное удаление куков**               | Logout → проверить куки                                | ✅ Куки удалены (Expires: 1970)                        |

## Лучшие практики безопасности

1. **HTTPS везде**: TLS 1.2/1.3 обязателен
2. **Валидация входных данных**: На сервере, не только на клиенте
3. **Параметризованные запросы**: Защита от SQL-инъекций
4. **Безопасные алгоритмы**: AES-256, SHA-256, bcrypt
5. **Куки с правильными флагами**: Secure, HttpOnly, SameSite
6. **2FA/MFA**: Для критичных операций
7. **Мониторинг и логирование**: Без логирования sensitive данных
8. **Обновления**: Своевременно обновлять зависимости
9. **Сканирование**: Регулярно проверяйте уязвимости (OWASP ZAP, Burp)
10. **Обучение команды**: Secure coding practices для разработчиков
