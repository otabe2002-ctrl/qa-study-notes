## Классы эквивалентности

### Определение

Разделение входных данных на группы, где каждая группа должна вести себя одинаково.

### Классы для поля логина

**Класс 1: Латинские буквы (позитивный)**

- `admin` (строчные буквы)
- `UserName` (смешанный регистр)
- **Ожидание**: Система принимает

**Класс 2: Цифры (позитивный)**

- `123456` (только цифры)
- `user2024` (буквы и цифры)
- **Ожидание**: Система принимает

**Класс 3: Специальные символы (негативный)**

- `user@mail` (символ @)
- `login#123` (символ #)
- **Ожидание**: Система отклоняет или обрабатывает по определению

**Класс 4: Символы кириллицы (негативный)**

- `пользователь` (строчные кириллические буквы)
- `Логин` (смешанный регистр кириллицы)
- **Ожидание**: Система отклоняет

**Класс 5: Пробелы и комбинированные символы (негативный)**

- `user name` (с пробелом)
- `user@123#abc` (комбинация букв, цифр и спецсимволов)
- **Ожидание**: Система отклоняет

### Преимущества

- Сокращение количества тестов
- Гарантированное покрытие всех типов входных данных
- Выявление граничных случаев

## Граничные значения

### Определение

Проверка поведения на границах диапазонов валидных значений.

### Пример: Лимит платежа

**Граничные точки:**

- Минимальное допустимое значение: 1 000
- Границы-1: 9 000, 10 000, 11 000 (для первого лимита)
- Границы-2: 49 000, 50 000, 51 000 (для второго лимита)
- Границы-3: 99 000, 100 000 (для максимума)

**Типичные значения:** 5 000, 30 000, 70 000

**Итоговый набор тестовых значений:**
`1 000, 5 000, 9 000, 10 000, 11 000, 30 000, 49 000, 50 000, 51 000, 70 000, 99 000, 100 000`

### Почему это важно

- На границах часто возникают ошибки
- Целочисленное переполнение (как в Ariane 5)
- Off-by-one ошибки

## Диаграммы переходов состояний

### Назначение

Визуализация всех возможных состояний системы и переходов между ними.

### Элементы диаграммы

- **Состояния**: Окружности с названиями
- **Переходы**: Стрелки между состояниями
- **Действия**: Подписи на стрелках (что вызывает переход)
- **Начальное состояние**: Отмечено специально
- **Конечное состояние**: Двойной круг

### Применение

- Проверка всех возможных переходов
- Выявление недостижимых состояний
- Тестирование невалидных переходов

## Pairwise Testing

### Определение

Метод генерирования тестовых случаев, где каждая пара значений двух параметров встречается хотя бы один раз.

### Пример: Кредитная заявка

**Параметры:**

- Возраст: 18–35, 36–55, 56–75 (3 варианта)
- Задолженности: да, нет (2 варианта)
- Тип трудовых отношений: Самозанятый, ИП, Бессрочный договор (3 варианта)
- Кредиты: да, нет (2 варианта)

**Полное количество комбинаций:** 3 × 2 × 3 × 2 = 36

**Pairwise (уникальные пары):** Около 9 тест-кейсов вместо 36

### Таблица pairwise тестирования

| Возраст | Задолженности | Тип отношений | Кредиты |
| -------------- | -------------------------- | ------------------------- | -------------- |
| 18–35         | да                       | Самозанятый    | да           |
| 18–35         | нет                     | Бессрочный      | нет         |
| 18–35         | да                       | ИП                      | да           |
| 36–55         | нет                     | ИП                      | да           |
| 36–55         | да                       | Самозанятый    | нет         |
| 36–55         | да                       | Бессрочный      | да           |
| 56–75         | да                       | Бессрочный      | да           |
| 56–75         | да                       | ИП                      | нет         |
| 56–75         | нет                     | Самозанятый    | да           |

### Преимущества

- Сокращение количества тестов на 80%+
- Гарантированное покрытие всех пар
- Экономия времени и ресурсов

## Тестовые туры (Test Tours)

### Определение

Исследовательское тестирование, где тестировщик целенаправленно изменяет поведение для имитации различных типов пользователей.

### Тур №1: "Нетерпеливый пользователь"

**Суть**: Быстро кликает, быстро прокручивает контент

**Гипотеза дефектов:**

- Race conditions и мультизагрузки
- Пропадание или дублирование элементов
- Неработающие ссылки на не полностью загруженной странице
- Потеря контекста

**Действия:**

1. Быстро прокрутить вверх-вниз несколько раз
2. Последовательно кликать на все интерактивные элементы
3. Открыть несколько ссылок одновременно
4. Быстро переходить между разделами
5. Проверить консоль браузера (F12)

### Тур №2: "Неуверенный пользователь"

**Суть**: Постоянно возвращается назад, перепроверяет информацию

**Гипотеза дефектов:**

- Проблемы с историей браузера
- Потеря заполненных данных
- Несоответствие состояния между клиентом и сервером
- Проблемы с кэшем

**Действия:**

1. Открыть раздел
2. Нажать "Назад" браузера 3-5 раз подряд
3. Проверить загрузку элементов
4. Нажать "Вперед" 3-5 раз
5. Открыть несколько ссылок и тестировать навигацию

### Сравнение туров

| Параметр                 | "Нетерпеливый"      | "Неуверенный"                   |
| -------------------------------- | ------------------------------- | ------------------------------------------ |
| **Фокус**             | Быстрые действия | Рефлексивные действия  |
| **Направление** | Вперед/вглубь       | Назад/проверка                |
| **Дефекты**         | Race conditions                 | История, состояние, кэш |
| **Сценарий**       | Спешит                    | Сомневается                     |

### Синергия туров

Вместе они дают полную картину реального поведения пользователей и покрывают противоположные сценарии.

## Матрица тестового покрытия

### Назначение

Классификация проверок по важности для определения приоритетов.

### Уровни критичности

**КРИТИЧЕСКИЕ (Tier 1) - 50%**

- Доступность и загрузка сайта
- Основная навигация и ссылки
- Корректность контента
- Базовая визуализация
- Кодировка и параметры
- Мультимедиа и интерактивность

**ВАЖНЫЕ (Tier 2) - 33%**

- Кроссбраузерность
- Адаптивность устройств
- Производительность
- Дополнительная навигация
- Совместимость ОС и устройств
- Элементы интерфейса

**ВСПОМОГАТЕЛЬНЫЕ (Tier 3) - 16%**

- Оптимизация производительности
- SEO и расширенные функции
- Экстремальные нагрузки

## Практические рекомендации

1. **Начните с классов эквивалентности и граничных значений** - это базовые техники
2. **Используйте диаграммы состояний** для сложных бизнес-логик
3. **Применяйте pairwise** при большом количестве параметров
4. **Проводите тестовые туры** дополнительно к формальным тест-кейсам
5. **Создайте матрицу покрытия** для управления приоритетами
