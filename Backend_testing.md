## REST API и HTTP методы

### Определение REST API

**REST (Representational State Transfer)** - архитектурный стиль для построения распределённых систем через протокол HTTP.

**Основная идея**: Работа с ресурсами (пользователи, товары, заказы) через стандартные HTTP-методы.

### Основные HTTP методы

#### GET

- **Использование**: Получение данных о ресурсе или коллекции
- **Особенность**: Не изменяет состояние, безопасный
- **Пример**: `GET /users` → получить список пользователей
- **Статусы**: 200 (OK), 404 (Not Found), 500 (Server Error)

#### POST

- **Использование**: Создание нового ресурса
- **Особенность**: Меняет состояние, данные в теле запроса
- **Пример**: `POST /users` → создать нового пользователя
- **Статусы**: 201 (Created), 400 (Bad Request), 409 (Conflict)

#### PUT

- **Использование**: Полное обновление существующего ресурса
- **Особенность**: Заменяет весь ресурс
- **Пример**: `PUT /users/123` → полностью обновить пользователя с ID 123
- **Статусы**: 200 (OK), 204 (No Content), 404 (Not Found)

#### PATCH

- **Использование**: Частичное обновление ресурса
- **Особенность**: Меняет только указанные поля
- **Пример**: `PATCH /users/123` → обновить только имя пользователя
- **Статусы**: 200 (OK), 204 (No Content)

#### DELETE

- **Использование**: Удаление ресурса
- **Особенность**: Необратимо
- **Пример**: `DELETE /users/123` → удалить пользователя с ID 123
- **Статусы**: 204 (No Content), 404 (Not Found)

## JSON и JSON Schema

### Структура JSON

#### Объекты (Objects)

- Неупорядоченный набор пар "ключ: значение"
- Заключены в фигурные скобки `{}`

#### Массивы (Arrays)

- Упорядоченный список значений
- Заключены в квадратные скобки `[]`

### Пример JSON

```json
{
  "api": "v.2.7",
  "user": {
    "id": 1024,
    "name": "Jane Doe",
    "roles": ["admin", "editor"],
    "active": true
  }
}
```

### JSON Schema

**Назначение**: Описание и валидация структуры JSON-документов

**Применение**:

- Валидация: Убедиться, что данные соответствуют ожиданиям
- Документация: Описать структуру для других разработчиков
- Тестирование: Автоматически генерировать тестовые данные
- Безопасность: Отсеивать некорректные данные на входе

### Пример JSON Schema

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "name": {
      "type": "string",
      "description": "Полное имя пользователя"
    },
    "age": {
      "type": "integer",
      "description": "Возраст в годах",
      "minimum": 0
    },
    "isStudent": {
      "type": "boolean",
      "description": "Является ли пользователь студентом"
    }
  },
  "required": ["name", "age"]
}
```

## Сериализация и десериализация

### Определения

**Сериализация**: Преобразование объекта из памяти в строку JSON для передачи/сохранения

**Десериализация**: Преобразование JSON-строки обратно в объект для работы в коде

### Пример сериализации (Python)

```python
import json

user_dict = {
    "name": "Иван",
    "age": 30,
    "isAdmin": False
}

# Превращаем словарь в строку JSON
json_string = json.dumps(user_dict)
print(json_string)
# Вывод: {"name": "Иван", "age": 30, "isAdmin": false}
```

### Пример десериализации (JavaScript)

```javascript
const jsonFromServer = '{"name":"Иван","age":30,"isAdmin":false}';

// Превращаем строку обратно в объект
const userObject = JSON.parse(jsonFromServer);
console.log(userObject.name);   // Вывод: Иван
console.log(userObject.age);    // Вывод: 30
```

## Сравнение API

### Reqres API (Учебное API)

**Ссылка**: https://reqres.in/

**Назначение**: Демонстрационное API для обучения основам REST

**Основные эндпоинты**:

- `GET /api/users` → список пользователей с пагинацией
- `GET /api/users/{id}` → получить конкретного пользователя
- `POST /api/users` → создать нового пользователя
- `PUT/PATCH /api/users/{id}` → обновить пользователя
- `DELETE /api/users/{id}` → удалить пользователя
- `POST /api/register` → регистрация (только eve.holt@reqres.in)
- `POST /api/login` → вход (только eve.holt@reqres.in)
- `GET /api/users?delay=3` → эмуляция задержки ответа

**Особенности**:

- ✅ Только JSON
- ✅ Тестовые данные (заранее заданные пользователи)
- ✅ Пагинация (page, per_page)
- ✅ Rate limiting: 60 запросов в минуту с IP

### Auth0 Authentication API

**Ссылка**: https://dev.auth0.com/docs/api/authentication

**Назначение**: Реализация OAuth 2.0 и OIDC протоколов для аутентификации

**Основные эндпоинты**:

- `GET /authorize` → аутентификация и получение кода/токена (редирект)
- `POST /oauth/token` → обмен кода на токены доступа
- `GET /userinfo` → получить профиль пользователя через Access Token
- `POST /dbconnections/signup` → регистрация в БД
- `POST /dbconnections/change_password` → смена пароля
- `GET /v2/oauth/logout` → выход

**Особенности**:

- ✅ OAuth 2.0 / OIDC (стандартные протоколы)
- ✅ Редиректы и кода авторизации
- ✅ JWT токены (ID Token, Access Token)
- ✅ Гибкие методы аутентификации (социальный вход, БД, смена пароля)

### Таблица сравнения

| Критерий                     | Reqres                             | Auth0                                   |
| ------------------------------------ | ---------------------------------- | --------------------------------------- |
| **Тип**                     | Учебное демо-API        | Production аутентификация |
| **Подход**               | CRUD операции              | OAuth 2.0 / OIDC стандарты     |
| **Форматы**             | JSON                               | JSON + редиректы               |
| **Авторизация**     | Простая (есть ключ) | Сложная (OAuth 2.0)              |
| **Использование** | Обучение                   | Production приложения         |

## ID Token (Пример из Auth0)

### Структура в декодированном виде

```json
{
  "iss": "https://dev-example.auth0.com/",
  "sub": "auth0|123456",
  "aud": "YOUR_CLIENT_ID",
  "exp": 1311281970,
  "iat": 1311280970,
  "name": "Иван Петров",
  "given_name": "Иван",
  "family_name": "Петров",
  "gender": "male",
  "birthdate": "1990-10-31",
  "email": "ivan.petrov@example.com",
  "email_verified": true,
  "picture": "https://example.com/avatar.jpg"
}
```

### Ключи и типы данных

| Ключ           | Тип  | Значение     | Описание                                                                  |
| ------------------ | ------- | -------------------- | --------------------------------------------------------------------------------- |
| `iss`            | string  | URL издателя | Сервер авторизации, выпустивший токен            |
| `sub`            | string  | auth0\|123456        | Уникальный ID пользователя                                  |
| `aud`            | string  | CLIENT_ID            | Приложение, для которого предназначен токен |
| `exp`            | integer | 1311281970           | Unix timestamp истечения токена                                    |
| `iat`            | integer | 1311280970           | Unix timestamp выпуска токена                                        |
| `email`          | string  | ivan@example.com     | Email пользователя                                                    |
| `email_verified` | boolean | true                 | Подтверждён ли email                                                 |

## Инструменты тестирования (Postman)

### Основные возможности

1. **Коллекции запросов**

   - Группировка запросов по функциональности
   - Организация окружений
   - Переиспользование переменных
2. **Глобальные переменные**

   - Хранение API ключей, токенов, base URL
   - Переиспользование в запросах
   - Безопасное хранение credentials
3. **GET-запрос с проверками**

   ```
   GET /users/2
   Response:
   {
     "data": {
       "id": 2,
       "email": "janet.weaver@reqres.in"
     }
   }
   ```
4. **POST-запрос на авторизацию**

   ```
   POST /login
   Body: {
     "email": "eve.holt@reqres.in",
     "password": "cityslicka"
   }
   Response: {
     "token": "QpwL5tke4Pj6/cVf2J..."
   }
   ```
5. **PUT-запрос на обновление**

   ```
   PUT /users/2
   Body: {
     "name": "morpheus",
     "job": "zion resident"
   }
   ```
6. **Test Runner**

   - Запуск всех тестов в коллекции
   - Формирование отчётов
   - Интеграция с CI/CD

### Лучшие практики

1. Организуйте запросы в коллекции по функциональности
2. Используйте переменные для переиспользуемых значений
3. Создавайте pre-request scripts для подготовки данных
4. Добавляйте test scripts для проверки ответов
5. Версионируйте коллекции в git
6. Документируйте сложные сценарии
7. Используйте окружения для разных стендов (dev, staging, prod)
