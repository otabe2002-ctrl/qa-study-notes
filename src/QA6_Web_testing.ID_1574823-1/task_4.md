# **HTTPS / TLS 1.3 Handshake — Техническая спецификация**

# Установление HTTPS-соединения (TLS 1.3)

## 1. Предварительные условия

### 1.1 DNS-запрос

Клиент выполняет DNS-запрос и получает A/AAAA-запись сервера:

* **A** — запись с IPv4-адресом
* **AAAA** — запись с IPv6-адресом

**Цель:** определить IP-адрес сервера.

**Риск:** без защищённого DNS возможна подмена ответа (DNS spoofing).

---

### 1.2 Установка TCP-соединения

Выполняется **3-way handshake** (трёхстороннее рукопожатие TCP).

Уровень 4 модели OSI — транспортный.

* **SYN** — запрос на установление соединения
* **SYN-ACK** — подтверждение и согласие установить соединение
* **ACK** — финальное подтверждение

**Цель:** создать надёжный канал передачи данных.

**Риск:** TCP устраняет проблемы потери и порядка пакетов, но не защищает от перехвата.

---

### 1.3 (Опционально) ECH

Если используется **ECH (Encrypted Client Hello)** — зашифрованная версия ClientHello,

клиент уже имеет публичный ключ сервера для шифрования SNI.

* **SNI (Server Name Indication)** — расширение TLS, позволяющее указать имя хоста при handshake.

Публичный ключ для ECH обычно передаётся через DNS (HTTPS-запись).

**Цель:** скрыть имя запрашиваемого сайта.

**Риск:** без ECH наблюдатель видит, к какому домену происходит подключение.

---

## 2. ClientHello (начало TLS-handshake)

Клиент отправляет:

* версию **TLS** (например, TLS 1.3 — версия протокола шифрования);
* список **cipher suites** (наборов криптографических алгоритмов);
* `client random` (случайное число клиента);
* расширения:
  * **SNI** — имя сервера;
  * **ALPN (Application-Layer Protocol Negotiation)** — согласование HTTP/1.1 или HTTP/2;
  * поддерживаемые группы для ECDHE.

**Цель:** инициировать согласование параметров шифрования.

**Риск:** предотвращается использование неподдерживаемых или устаревших алгоритмов.

---

## 3. ServerHello + сертификат

Сервер:

* выбирает версию TLS;
* выбирает одну cipher suite;
* отправляет `server random` (случайное число сервера);
* передаёт **сертификат X.509**(стандартный формат цифрового сертификата с публичным ключом сервера);
* передаёт параметры обмена ключами —  **ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)** .

**Цель:** определить параметры шифрования и подтвердить личность сервера.

**Риск:** без сертификата возможна подмена сервера (impersonation).

---

## 4. Проверка сертификата клиентом

Браузер проверяет:

* подпись **CA (Certificate Authority)** — удостоверяющего центра;
* соответствие **CN (Common Name)** / **SAN (Subject Alternative Name)** домену;
* срок действия сертификата;
* отзыв через:
  * **OCSP (Online Certificate Status Protocol)** — онлайн-проверка статуса;
  * **CRL (Certificate Revocation List)** — список отозванных сертификатов.

**Цель:** убедиться, что сервер подлинный.

**Риск:** защита от MITM-атак (человек посередине).

---

## 5. Обмен ключами (ECDHE)

Клиент и сервер обмениваются временными публичными ключами (ephemeral keys)

и вычисляют общий секрет.

Из этого секрета формируются симметричные сеансовые ключи.

* Даже при перехвате трафика вычислить общий секрет невозможно.
* Обеспечивается **Perfect Forward Secrecy** — компрометация долгосрочного ключа сервера не раскрывает прошлые сессии.

**Цель:** создать общий симметричный ключ.

**Риск:** исключается возможность расшифровки трафика при его перехвате.

---

## 6. Finished-сообщения

Обе стороны отправляют зашифрованные сообщения `Finished`,

содержащие хэш всех предыдущих этапов handshake.

**Цель:** подтвердить целостность рукопожатия.

**Риск:** предотвращается скрытая модификация параметров соединения.

---

## 7. Передача HTTP-данных поверх TLS (HTTPS)

После завершения handshake начинается передача прикладных данных.

* Используются симметричные алгоритмы:
  * **AES-GCM**
  * **ChaCha20-Poly1305**
* Применяется **AEAD (Authenticated Encryption with Associated Data)** — режим шифрования, обеспечивающий конфиденциальность и целостность.

**Цель:** защитить передаваемые данные.

**Риск:** предотвращается прослушивание, подмена данных и кража cookie/токенов.

Браузер не показал, какое именно имя сайта (SNI), какой набор шифров предлагался и какой был выбран, как именно происходил обмен ключами и вычисление общего секрета, что сервер подписывал хеш переговоров и что браузер эту подпись проверял, а также не показал финальную проверку целостности всего рукопожатия (Finished) и не отобразил факт проверки сертификата на отзыв через OCSP или CRL.

## **Что скрыто:**

* **Криптографическая подпись** (Целостность)

Браузер берет открытый ключ Центра сертификации (CA) и расшифровывает подпись, которой подписан сертификат сайта. Затем он сравнивает расшифрованные данные с реальными данными сертификата

* **Цепочка доверия** (Chain of Trust)

Браузер проверяет, ведет ли цепочка от сертификата сайта через промежуточные сертификаты к  **корневому сертификату** , который уже встроен в браузер/операционную систему и помечен как "доверенный"

* **Статус отзыва** (CRL)

Браузер проверяет, не аннулирован ли сертификат досрочно (например, при утечке ключа). Современные браузеры скачивают и кэшируют специальные списки отозванных сертификатов (CRL), которые публикуют Центры сертификации

* **Прозрачность сертификатов** (Certificate Transparency)

Браузер проверяет, есть ли в сертификате специальные метки (SCT), доказывающие, что он был записан в публичные, неизменяемые логи. Это защищает от ситуации, когда ЦС тайно выпустил сертификат на чужой домен

## 1xx — Информационные

> Сервер принял запрос и продолжает обработку. Пользователь обычно ничего не видит.

| Код                  | Что видит пользователь | Что происходит на самом деле                                        | Как тестировать                                                 | Куда эскалировать |
| ----------------------- | ------------------------------------------ | ------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | --------------------------------- |
| 100 Continue            | Ничего                               | Сервер разрешил продолжить отправку тела запроса | Проверять при загрузке больших файлов (POST) | Бэкенд                      |
| 101 Switching Protocols | Ничего                               | Переключение протокола (например, на WebSocket)              | Проверить успешное подключение WebSocket          | Бэкенд / DevOps             |

## 2xx — Успешные ответы

> Запрос обработан корректно.

| Код         | Что видит пользователь                               | Что происходит на самом деле | Как тестировать                                                     | Куда эскалировать |
| -------------- | ------------------------------------------------------------------------ | ---------------------------------------------------- | --------------------------------------------------------------------------------- | --------------------------------- |
| 200 OK         | Страница или данные отображаются            | Запрос успешно выполнен         | Проверить корректность отображаемых данных | Фронтенд / Бэкенд   |
| 201 Created    | Успешное создание (регистрация, запись) | Создан новый ресурс                 | Проверить запись в БД                                           | Бэкенд                      |
| 204 No Content | Нет видимого ответа                                     | Успешно, но без тела ответа    | Проверить корректную реакцию UI                         | Фронтенд                  |

## 3xx — Перенаправления

> Сервер сообщает, что ресурс находится в другом месте или используется кэш.

| Код                | Что видит пользователь            | Что происходит на самом деле | Как тестировать                                  | Куда эскалировать |
| --------------------- | ----------------------------------------------------- | ---------------------------------------------------- | -------------------------------------------------------------- | --------------------------------- |
| 301 Moved Permanently | Перенаправление на другой URL  | Постоянный редирект                | Проверить корректность нового URL   | DevOps / Бэкенд             |
| 302 Found             | Временный редирект                   | Временное перенаправление    | Проверить логику перехода               | Бэкенд                      |
| 304 Not Modified      | Страница загружается из кэша | Используется кэш браузера     | Очистить кэш и сравнить поведение | Фронтенд / Бэкенд   |

## 4xx — Ошибки клиента

> Запрос некорректен или нет доступа.

| Код                     | Что видит пользователь          | Что происходит на самом деле                | Как тестировать                             | Куда эскалировать |
| -------------------------- | --------------------------------------------------- | ------------------------------------------------------------------- | --------------------------------------------------------- | --------------------------------- |
| 400 Bad Request            | Сообщение о неверных данных | Ошибка в формате запроса                       | Отправить невалидные данные      | Бэкенд                      |
| 401 Unauthorized           | Требуется авторизация           | Нет валидного токена                              | Проверить работу авторизации    | Бэкенд                      |
| 403 Forbidden              | Доступ запрещён                       | Недостаточно прав                                   | Проверить роли пользователя      | Бэкенд                      |
| 404 Not Found              | Страница не найдена                | Ресурс отсутствует                                 | Перейти по неверному URL                | Фронтенд / Бэкенд   |
| 405 Method Not Allowed     | Метод запрещён                         | Использован неверный HTTP-метод             | Изменить метод запроса                | Бэкенд                      |
| 409 Conflict               | Конфликт данных                       | Дублирование или конфликт состояния | Попробовать создать дубликат    | Бэкенд                      |
| 415 Unsupported Media Type | Неподдерживаемый формат       | Неверный тип файла или Content-Type              | Загрузить файл другого формата | Бэкенд                      |

## 5xx — Ошибки сервера

> Сервер не смог корректно обработать запрос.

| Код                    | Что видит пользователь | Что происходит на самом деле       | Как тестировать                                | Куда эскалировать |
| ------------------------- | ------------------------------------------ | ---------------------------------------------------------- | ------------------------------------------------------------ | --------------------------------- |
| 500 Internal Server Error | Ошибка сервера                | Исключение или ошибка в коде       | Повторить запрос, проверить логи | Бэкенд                      |
| 502 Bad Gateway           | Ошибка шлюза                    | Один сервис не отвечает другому | Проверить доступность API                | DevOps                            |
| 503 Service Unavailable   | Сервис недоступен          | Перегрузка или техработы             | Проверить повторный доступ           | DevOps                            |
| 504 Gateway Timeout       | Таймаут ответа                | Сервер не получил ответ вовремя | Проверить длительные запросы       |                                   |
