Я предполагаю, что Telegram использует **гибридную архитектуру**, сочетающую **нереляционные (NoSQL)** и **реляционные базы данных**.

* NoSQL: по причине масштабируемости, каждый пользователь может иметь совершенно разные параметры, и у всех чатов своя структура засчет гибкости.
* Реляционная: хранение базовых данных аккаунтов, ключей

| Вид сервиса                                           | БД                   | Какие данные использует                                                                |
| --------------------------------------------------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------- |
| Файловое хранилище                             | NoSQL                  | бинарные данные, которые не укладываются в строки таблиц    |
| Отправка сообщений (История чатов) | Реляционная | Сообщения четко привязаны к чату, имеют дату и автора           |
| Криптовалюта Toncoin и платежии            | Блокчейн(TON)  | Балансы аккаунтов в сети TON, история транзакций кошельками |

**Идея тест-кейса 1:**

Цель: Проверить, как хранилище обрабатывает массовую подписку/ отписку

Сценарий:

1. Имитировать добавление 1 миллиона новых подписчиков в канал
2. Запустить проверку принадлежности к каналу для случайных пользователей
3. Проверить, как хранилище справляется с резкими изменениями количества подписчиков канала и не "тормозит" ли при этом проверку подписок для обычных пользователей.
4. Не дожидаясь окончания подписки, запускаем процесс, который начинает удалять часть пользователей
5. В самый пик нагрузки отправляем несколько дублирующихся команд подписки для одних и тех же пользователей.
