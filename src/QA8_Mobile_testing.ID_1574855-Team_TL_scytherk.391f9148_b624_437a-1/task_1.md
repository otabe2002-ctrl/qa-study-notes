#### Рекомендации по выбору сервиса

1. Основной выбор: **TestFlight**

   - Официальное решение Apple, интеграция с Xcode и App Store Connect;
   - Удобная установка и автообновления для тестировщиков, встроенный фидбек;
   - Не нужно платить отдельно за сервис — только стандартная подписка Apple Developer.
2. Дополнительный вариант: Firebase App Distribution, если планируется Android‑версия и хотите единый процесс для обеих платформ.
3. Эпизодический вариант: Diawi — для быстрых внутренних проверок (например, дать продукту/дизайнеру "посмотреть сборку прямо сейчас"), а не как основной канал беты.

### Утечка памяти

> Утечка памяти в контексте мобильного приложения.

В iOS за уборку отвечает специальный механизм — ARC (Automatic Reference Counting).
Переводится как «Автоматический подсчет ссылок».

***Как работает ARC?*:**
У каждого объекта есть счетчик ссылок, если на объект никто не ссылается, то ARC удаляет объект из памяти, если есть хотя бы одна активная ссылка, то она остаётся в памяти и занимает её, а ARC из-за активной ссылки на объект не может удалить его(объект) из памяти.

> **Почему возникает утечка? (Retain Cycle)**

Допустим, в  финансовом приложении есть Экран профиля и Аватарка пользователя.
В экран профиля показывается аватарка пользователя.
Как это работает в коде (упрощенно):

```
Есть объект ProfileScreen (Экран).
Есть объект UserAvatar (Картинка).
```

Чтобы экран мог управлять картинкой (обновить, спрятать), он должен на нее ссылаться.
`ProfileScreen -> ссылается на -> UserAvatar`
Часто делают так, чтобы картинка могла сообщить экрану: "По мне кликнули!".
Для этого UserAvatar должен знать, кто его хозяин.
`UserAvatar -> теперь ссылается на -> ProfileScreen`
*Получается замкнутый круг:*
Экран держит Картинку -> Картинка держит Экран.
*Что происходит, когда пользователь уходит с экрана?*
Пользователь нажал "Назад". Экран должен исчезнуть и очистить память.
Но! Экран не может исчезнуть, потому что его держит Картинка.
А Картинка не может исчезнуть, потому что ее держит Экран.

> УТЕЧКА. Оба объекта зависли в памяти мертвым грузом.

**Как это убивает приложение?**
Представь, что пользователь 50 раз зашел в профиль и вышел.
Каждый раз создается новая пара ProfileScreen и UserAvatar, которая навсегда залипает в памяти (ведь они держат друг друга).
Через 50 раз памяти не остается.
Телефон начинает тормозить, греться, а потом просто закрывает приложение.

Прежде чем давать рекомендации, надо понять, что у любого приложения есть жизненный цикл, если разбирать то в нашей ситуации в упрощенном виде нам важны такие состояния:

> ***Состояния жизненного цикла iOS‑приложения***

- Foreground (active) — приложение на экране, пользователь с ним взаимодействует.
- Background — приложение не на экране, но ещё может немного выполнять код (завершение задач, фоновые операции).
- Suspended — код не выполняется, но объекты приложения всё ещё в памяти.
- Terminated / Not running — процесс выгружен, память очищена.

> Сообщение коллеге:

```Привет!

- В активном режиме (foreground) утечки видны сильнее всего: приложение постепенно начинает потреблять всё больше памяти, может тормозить, лагать при навигации между экранами, а в какой‑то момент iOS может просто завершить приложение из‑за нехватки памяти, и для пользователя это выглядит как «краш».

- В фоне (background) и особенно в состоянии suspended объекты, которые утекли, продолжают занимать память, пока приложение ещё находится в памяти устройства. Если утечек много, система может выгрузить приложение из памяти раньше, чем хотелось бы, и тогда при возвращении пользователь увидит «холодный старт» вместо быстрого возобновления с того же места.

- При частых переходах между экранами и режимами (foreground ↔ background) небольшие утечки могут «накапливаться»: каждый раз создаются новые объекты (экраны, контроллеры, модели), но старые не освобождаются из‑за retain‑циклов, и общее потребление памяти растёт от сессии к сессии.

- После полного завершения приложения (terminated) вся память, включая утёкшие объекты, системой освобождается, но это не решает проблему: при следующем запуске утечки начнут накапливаться заново, и пользователи по‑прежнему будут сталкиваться с тормозами и неожиданными вылетами.

Поэтому нам важно регулярно мониторить использование памяти (инструменты Xcode) и устранять retain‑циклы, чтобы приложение было стабильно как в активном режиме, так и в фоне.
```
