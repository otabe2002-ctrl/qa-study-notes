## 1. Кодирование, хеширование и шифрование

### Определения и различия

| Параметр                 | Кодирование                                                 | Хеширование                                                  | Шифрование                                                                   |
| -------------------------------- | ---------------------------------------------------------------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **Назначение**   | Представление данных в другом формате | Создание уникального отпечатка данных | Защита данных от несанкционированного доступа |
| **Обратимость** | Да (двусторонний процесс)                         | Нет (одностороннее)                                     | Да (с ключом)                                                                 |
| **Ключ**               | Не требуется                                                | Не требуется                                                 | Требуется                                                                     |
| **Пример**           | Base64, URL-кодирование                                     | MD5, SHA, bcrypt                                                        | AES, RSA, DES                                                                          |

### Примеры использования в веб-приложении

1. **Кодирование**: Base64 для передачи бинарных данных в JSON API. Пример: `http://example.com/api?data=SGVsbG8gV29ybGQ=` (закодированное "Hello World").
2. **Хеширование**: Хранение пароля пользователя в БД. Пример: пароль "password123" хранится как `$2b$12$...` (bcrypt-хеш), при входе система хеширует введенный пароль и сравнивает хеши.
3. **Шифрование**: Защита платежных данных (номер карты, CVV). Пример: номер карты `4111 1111 1111 1111` шифруется AES-256 перед передачей на сервер платежей.

## 2. Симметричное vs. Асимметричное шифрование

### Аналогия для менеджера проекта

**Симметричное шифрование** = **Обычный замок на двери дома**

- Есть один ключ, который открывает и закрывает замок.
- Все, кто должен войти, должны иметь одинаковый ключ.
- Быстро, эффективно, но нужно как-то передать ключ другим людям (проблема распределения ключей).

**Асимметричное шифрование** = **Система почтовых ящиков с выключателем**

- У каждого есть два ключа: **публичный** (как адрес почтового ящика — все его знают) и **приватный** (только хозяин имеет).
- Любой может положить письмо в ящик (зашифровать публичным ключом), но открыть его может только хозяин приватным ключом.
- Медленнее, но безопаснее — не нужно ничего передавать секретно.

### Сравнительная таблица

| Параметр                 | Симметричное                                                                            | Асимметричное                                                           |
| -------------------------------- | --------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| **Определение** | Один и тот же ключ для шифрования и расшифровки              | Два разных ключа: публичный и приватный             |
| **Аналогия**       | Обычный замок на двери                                                           | Почтовый ящик + секретный ключ                              |
| **Плюсы**             | Быстро, простое, малый размер ключей                                  | Безопасная передача, аутентификация, подписи  |
| **Минусы**           | Проблема распределения ключей, сложно масштабировать | Медленное, большие ключи, сложнее реализовать |
| **Примеры**         | AES, 3DES, ChaCha20                                                                                 | RSA, ECDSA, Diffie-Hellman                                                           |

## 3. "Черный" и "белый" списки алгоритмов

### "Черный список" — НЕБЕЗОПАСНЫЕ алгоритмы

#### Симметричное шифрование

| Алгоритм                         | Слабые точки / Атаки                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **DES (Data Encryption Standard)** | • **Слишком короткий ключ** (56 бит) — может быть перебран брутфорсом за часы на современном ПК. <br />**• Предсказуемые паттерны в шифре.**<br />• **Уязвимость EFF DES Cracker** — специализированное оборудование взламывало за < 1 часа.                                                                                                                                                          |
| **RC4 (Rivest Cipher 4)**          | •**Ошибки в инициализации** — позволяют восстановить ключ со слабыми значениями. <br />• **Ненулевые вероятности** в потоке ключей приводят к компрометации (тесты показали, что первые байты отличаются от случайных). <br />• **Уязвимость для атак на SSL/TLS** — отпечаток трафика позволяет угадать части сообщений. |
| **ECB (Electronic Codebook mode)** | •**Одинаковые блоки открытого текста** дают одинаковые блоки шифра — видны паттерны. <br />• **Атака "Pattern analysis"** — например, шифрование всех одинаковых пикселей изображения дает одинаковые выходы, виден силуэт.                                                                                                                                                                                    |

#### Хеширование

| Алгоритм                          | Слабые точки / Атаки                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **MD5 (Message Digest 5)**          | •**Столкновения (collisions)** — найдены два разных файла с одинаковым хешем (криптографическая стойкость нарушена). <br />• **Атака на сертификаты** — использовалась для подделки SSL сертификатов (MD5 сертификаты были отозваны). <br />• **Быстрые хеширование позволяет ускоренный брутфорс** — можно перебирать миллиарды паролей в секунду. |
| **SHA-1 (Secure Hash Algorithm 1)** | •**Теоретические и практические коллизии** — в 2017 году Google показал коллизию SHA-1. <br />• **160-бит выход** — меньше, чем у SHA-256, слабее против атак "дней рождения". <br />• **Удаление из SSL сертификатов** — браузеры перестали доверять SHA-1 подписям с 2016 года.                                                                                                                                                        |
| **CRC32**                           | •**Не криптографическая функция** — предназначена для проверки ошибок, не защиты.<br />• **Тривиальный перебор** — 32-бит выход легко перебирается (2^32 = 4 млрд комбинаций). <br />• **Линейность** — можно манипулировать данными и подобрать нужный CRC.                                                                                                                                                                |

### "Белый список" — БЕЗОПАСНЫЕ алгоритмы

#### Симметричное шифрование

| Алгоритм                                         | Почему безопасен                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| -------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **AES (Advanced Encryption Standard) — Rijndael** | •**Утвержден правительством США** для данных "TOP SECRET". <br />• **Ключи 128, 192, 256 бит** — даже 128-бит невозможно перебрать на современной технике. <br />• **15 лет проверок**(с 2001) без выявленных критических уязвимостей. <br />• **Используется везде**: банки, гос. учреждения, военные. |
| **ChaCha20**                                       | • **Современная альтернатива AES** на основе работ Bernstein. <br />• **Более быстра на устройствах без AES-NI** (мобильные, старые ПК). <br />• **Использует более стойкую математику** против differential & linear cryptanalysis. <br />• **Принята как стандарт** в TLS 1.3 и WireGuard VPN.                                                |

#### Хеширование

| Алгоритм                             | Почему безопасен                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **SHA-256 (SHA-2 семейство)** | •**256-бит выход** — достаточно большой, чтобы защитить от коллизий на 2^128 попыток (экспоненциальное усиление). <br /> • **Коллизии не найдены** — несмотря на 20+ лет исследований. <br />• **Используется в Bitcoin** для критичных хешей. <br />• **Стандарт NIST** и одобрен ISO.                                                                     |
| **bcrypt**                             | •**Специально для паролей** — медленное хеширование (требует много процессорного времени). <br />• **Встроена "соль"** — защита от rainbow tables и одинаковых паролей. <br /> • **Адаптивный cost factor** — можно увеличивать сложность при росте мощности ПК.<br />  • **Используется повсеместно** в production: Django, Ruby, PHP. |

## 6. Проверка: Пользователь забыл разлогиниться на общественном компьютере

### Как это можно проверить?

1. **Имитация общественного компьютера**:

   - Откройте магазин на компьютере в режиме "приватного окна" браузера (как бы общественный ПК).
   - Выполните вход с учетной записью.
   - Закройте браузер **БЕЗ явного выхода** (logout).
2. **Проверка №1 — Восстановление сессии**:

   - Закройте браузер полностью.
   - Откройте браузер заново (или новое приватное окно).
   - Перейдите на сайт магазина.
   - Проверьте, остались ли куки сессии → возможен ли доступ в кабинет без входа?
3. **Проверка №2 — Проверка куков в DevTools**:

   - Нажмите F12 → Application → Cookies.
   - Посмотрите флаги `Secure`, `HttpOnly`, `SameSite`, время `Expires`.
4. **Проверка №3 — Перехват через proxy**:

   - Используйте Burp Suite.
   - Войдите в аккаунт.
   - Закройте браузер.
   - Откройте Burp Suite → смотрите, какие куки передавались.
5. **Проверка №4 — Попытка доступа со старой сессией**:

   - Сохраните cookie после входа.
   - Скопируйте значение `session_id`.
   - На другом браузере/компьютере подменьте cookie на сохраненное значение.
   - Проверьте, можете ли вы зайти в кабинет другого пользователя?

### Что должно происходить для защиты?

| Мера защиты                                                                  | Как проверить                                                                                      | Ожидаемое поведение                                                                                       |
| -------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **Автоматический logout при закрытии браузера** | Закрыть браузер, открыть — проверить доступ                               | После перезагрузки браузера — требуется повторный вход                      |
| **Флаг `HttpOnly` на куках сессии**                           | DevTools → Cookies → проверить флаг `HttpOnly`                                                | Cookie отмечен как `HttpOnly: ✓` — недоступен из JavaScript                                       |
| **Флаг `Secure` на куках сессии**                             | DevTools → Cookies → проверить флаг `Secure`                                                  | Cookie отмечен как `Secure: ✓` — передается только по HTTPS                                 |
| **Флаг `SameSite` (Strict/Lax)**                                           | DevTools → Cookies → проверить `SameSite`                                                         | Cookie отмечен как `SameSite=Strict` или `SameSite=Lax` — защита от CSRF                          |
| **Таймаут сессии (Session Timeout)**                                | Войти, оставить открытым браузер на 30 минут без активности    | Сессия истекает, требуется повторный вход при следующем действии    |
| **Проверка User-Agent при восстановлении сессии** | Войти на Windows, попробовать использовать cookie на Mac                       | Система обнаружит смену User-Agent, потребует повторной аутентификации |
| **Проверка IP-адреса сессии**                                | Войти с IP 192.168.1.1, попробовать использовать cookie с другого IP      | При смене IP требуется переаутентификация                                                |
| **CSRF-токены в формах**                                            | Попробовать отправить форму со старой сессией без CSRF-токена | Запрос отклоняется с ошибкой 403 Forbidden                                                         |
| **Явное удаление куков при logout**                         | Выполнить logout через API, проверить куки                                          | Куки удалены (Expires: Thu, 01 Jan 1970) или Set-Cookie с пустым значением                    |
