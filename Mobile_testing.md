## Мобильные ОС и платформы

### iOS

- **Язык разработки**: Swift (или Objective-C)
- **IDE**: Xcode
- **Эмулятор**: iOS Simulator
- **App Store**: Официальное хранилище приложений

### Android

- **Язык разработки**: Kotlin (или Java)
- **IDE**: Android Studio
- **Эмулятор**: Android Emulator
- **App Store**: Google Play (или альтернативные)

## TestFlight и Firebase App Distribution

### TestFlight (Apple)

**Стоимость**:

- Для разработчиков: $99/год Apple Developer Account
- Для тестировщиков: Бесплатное участие

**Простота настройки для разработчика**:

1. Apple Developer Account
2. Настройка в App Store Connect
3. Загрузка сборки из Xcode
4. Быстрая бета-модерация Apple

**Простота установки для тестировщика**:

1. Скачать TestFlight из App Store
2. Принять приглашение по почте
3. Автоматические обновления

**Ограничения**:

- До 10 000 тестировщиков на сборку
- Требуется Apple ID
- Требуется Apple Developer Account

### Google Firebase App Distribution

**Стоимость**:

- Базовый Firebase: Бесплатен
- Платные тарифы: Зависят от других сервисов
- Хранилище > 1 ГБ: Может потребоваться Blaze тариф

**Простота настройки для разработчика**:

1. Подключить Firebase к проекту
2. Автоматизация UDID (Firebase автоматически регистрирует)
3. Разработчик не собирает UDID вручную

**Простота установки для тестировщика**:

1. Получить письмо/ссылку
2. Принять приглашение через Google-аккаунт
3. Скачать приложение

**Ограничения**:

- Бесплатный тариф (Spark): 1 ГБ общего хранилища
- Требует Google-аккаунта

### Diawi

**Стоимость**:

- Free: €0
- Starter: €2.99/месяц
- Premium: €29.99/месяц
- Enterprise: €299.99/месяц

**Простота настройки**:

- Залить .ipa в веб-форму
- Получить ссылку

**Простота установки**:

- Открыть ссылку
- Установить по инструкции

**Ограничения**:

- Лимит размера и времени жизни ссылки
- Больше для быстрых проверок, чем долгой беты
- Для финтеха: публичные ссылки не безопасны

### Рекомендации

1. **Основной выбор**: TestFlight (официальное решение Apple)
2. **Для Android**: Firebase App Distribution (единый процесс для обеих платформ)
3. **Эпизодический выбор**: Diawi (для быстрых дизайн-проверок)

## Внешние прерыватели (External Interruptions)

### Чек-лист тестирования

```
☐ Входящий звонок (обычный) во время подтверждения платежа
☐ Входящий звонок через мессенджер (WhatsApp/Telegram)
☐ Входящий SMS (код 2FA)
☐ Входящий звонок при биометрической аутентификации (Face ID)
☐ Блокировка экрана (кнопкой или таймаутом)
☐ Системные алерты (низкий заряд, нет сети, обновление iOS)
☐ Системный диалог разрешений (FaceID, Камера, Геолокация)
☐ Таймаут сессии / автоматический логаут
☐ Приложение убито в фоне во время платежа
☐ Резкое отключение/потеря сети (Wi-Fi → offline)
```

### Сценарий 1: Входящий звонок при платеже

**Ожидаемое поведение**:

- Приложение не падает
- Статус операции отображается корректно
- Нет двойного списания
- История показывает правильный результат

### Сценарий 2: Потеря сети во время платежа

**Ожидаемое поведение**:

- Приложение корректно сообщает об ошибке
- Статус однозначный (не выполнено)
- После восстановления нет двойного списания

### Сценарий 3: Блокировка экрана при заполнении формы

**Вариант А (заполнение)**:

- Введённые данные не потеряны или потеряны предсказуемо
- Не требуется повторная авторизация без причины

**Вариант B (ожидание ответа)**:

- Отображается актуальный и однозначный статус
- Нет неопределённости (списались деньги или нет)

## Утечки памяти (Memory Leaks)

### Как работает ARC (Automatic Reference Counting)

У каждого объекта есть счетчик ссылок:

- Если на объект никто не ссылается → ARC удаляет объект
- Если есть активная ссылка → объект остаётся в памяти

### Retain Cycle (Утечка памяти)

**Пример: Экран профиля с аватаркой**

```
ProfileScreen → ссылается на → UserAvatar
UserAvatar → ссылается на → ProfileScreen
↑_________________________________↓
       Замкнутый круг (утечка)
```

**Что происходит**:

1. Пользователь открывает профиль
2. Создаются объекты ProfileScreen и UserAvatar
3. Пользователь закрывает экран
4. Экран должен удалиться, но:
   - Экран не может удалиться, потому что его держит аватарка
   - Аватарка не может удалиться, потому что её держит экран
   - **УТЕЧКА**: Оба объекта зависли в памяти мертвым грузом

**Проблемы**:

- Приложение постепенно потребляет больше памяти
- Тормозит и греется
- iOS может закрыть приложение из-за нехватки памяти

### Состояния жизненного цикла iOS-приложения

- **Foreground (active)**: На экране, пользователь взаимодействует (утечки видны сильнее)
- **Background**: Не на экране, но код выполняется (утечки накапливаются)
- **Suspended**: Код не выполняется, объекты в памяти (утечки остаются)
- **Terminated**: Процесс выгружен, память очищена

### Влияние утечек в разных состояниях

**Foreground**:

- Видно сразу: тормозит, лагает при навигации
- iOS может закрыть приложение из-за нехватки памяти

**Background и Suspended**:

- Утечки накапливаются от сессии к сессии
- При возвращении пользователь видит "холодный старт" вместо быстрого возобновления

**Совет**: Мониторьте память в Xcode Tools и устраняйте retain-циклы

## Типы мобильных приложений

### Нативное приложение

**Суть**: Пишутся отдельно под каждую платформу (iOS на Swift, Android на Kotlin)

| Плюсы                                                    | Минусы                                      |
| ------------------------------------------------------------- | ------------------------------------------------- |
| Максимальная производительность | Большие затраты по бюджету |
| Полный доступ к API                              | Две кодовые базы                    |
| Лучший пользовательский опыт        | Долгий старт (две команды)   |

**Подходит для**: Приложения с высокими требованиями к производительности и нативному внешнему виду

### Кросс-платформенное приложение (Flutter, React Native)

**Суть**: Один код для обеих платформ, собирается в нативные приложения

| Плюсы                                                            | Минусы                                                    |
| --------------------------------------------------------------------- | --------------------------------------------------------------- |
| Дешевле чем два нативных                         | Зависимость от фреймворка                |
| Быстрее чем два нативных                         | Возможные задержки в поддержке       |
| Производительность близка к нативной | Некоторые API могут быть недоступны |

**Подходит для**: Стартапы с ограниченным бюджетом, которым нужно быстро выйти на обе платформы

### Гибридное приложение (Ionic, Cordova)

**Суть**: Веб-приложение (HTML, CSS, JS) в оболочке WebView

| Плюсы                | Минусы                                  |
| ------------------------- | --------------------------------------------- |
| Быстрый старт | UI хуже чем нативные           |
| Дешево              | Производительность ниже |
| Один код           | Ограничения плагинов       |

**Подходит для**: Простые приложения, где производительность не критична

### Веб-приложение (PWA)

**Суть**: Адаптированный под мобильные браузер сайт

| Плюсы                                 | Минусы                                         |
| ------------------------------------------ | ---------------------------------------------------- |
| Максимально быстро        | Ограниченный доступ к API         |
| Минимальный бюджет        | Push-уведомления нестабильны   |
| Очень просто обновлять | Фоновые процессы ограничены |

**Подходит для**: Информационные приложения, где нужен минимальный бюджет

## Рекомендации по выбору типа

**Для стартапа с ограниченным бюджетом и сроками 4 месяца**:

✅ **Выбор: Кросс-платформенное приложение (Flutter/React Native)**

**Обоснование**:

1. **Один код** → одна кодовая база, легче и быстрее разработать
2. **Функциональность** → Хороший доступ к GPS, push-уведомлениям, камере
3. **Время** → Можно выпустить MVP на обе платформы за 4 месяца
4. **Бюджет** → Дешевле чем два нативных приложения
5. **Масштабирование** → Один код легче поддерживать в будущем

**Потенциальные риски**:

- ⚠️ Производительность геолокации может быть ниже, чем на нативном
- ⚠️ Push-уведомления могут работать по-разному на iOS и Android
- ⚠️ Плагины для камеры и платежей могут быть нестабильны

**Фокус тестирования**:

1. Геолокация в реальном времени (разные устройства и ОС)
2. Push-уведомления в разных состояниях (фон, экран заблокирован)
3. Камера для сканирования промокодов (разные модели, освещение)
4. Платежные интеграции и обработка ошибок

## Практические рекомендации

1. **Выбирайте платформы по требованиям**, не по предпочтениям
2. **Тестируйте на разных устройствах** (iPhone 6-15, Samsung, Pixel)
3. **Проверяйте утечки памяти** регулярно (Xcode Instruments)
4. **Тестируйте прерывания** (звонки, SMS, потеря сети)
5. **Используйте правильный инструмент для бета-тестирования** (TestFlight для iOS, Firebase для Android)
6. **Документируйте риски** связанные с выбранным типом приложения
