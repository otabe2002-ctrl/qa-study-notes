## Архитектура веб-приложений

### Трёхзвенная архитектура

```
Client (Browser/Mobile)
    ↓
Презентационный слой (UI, CSS, JS)
    ↓
Прикладной слой (Бизнес-логика, Application Server)
    ↓
Слой данных (БД)
```

### Многозвенная архитектура (Microservices)

```
Client (Browser/Mobile)
    ↓
CDN / WAF
    ↓
Load Balancer
    ↓
Web Servers (Frontend)
    ↓
API Gateway / Auth Service
    ↓
Application Servers (Microservices: catalog, cart, payment, search)
    ↓
Cache (Redis) ←→ Database Cluster (Primary + replicas)
    ↓
External Services (Payment gateway, Email/SMS, 3rd-party APIs)
```

### Ключевые компоненты

| Компонент         | Назначение                                                                        | Типичные сбои                                                              |
| -------------------------- | ------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **CDN**              | Доставка статического контента с ближайшего узла | Неправильный кэш, потеря актуальности                 |
| **Load Balancer**    | Распределение запросов, отказоустойчивость           | Неправильное распределение, потеря соединений |
| **API Gateway**      | Централизованная точка входа для микросервисов    | Конфликты версий, потеря данных                             |
| **Microservices**    | Автономные сервисы (каталог, корзина, платежи)        | Временные отказы, несоответствие данных             |
| **Database Cluster** | Хранение данных (мастер-реплики)                                 | Потеря мастера, отставание репликации                 |
| **Cache (Redis)**    | Ускорение доступа к часто запрашиваемым данным     | Некорректные данные, потеря синхронизации         |

## Отказоустойчивость

### Горячий резерв (Hot Standby)

- **Готовность**: Мгновенная / минимальная
- **Синхронизация**: Постоянная (репликация, зеркалирование)
- **Время переключения (RTO)**: Низкое (секунды – минуты)
- **Потеря данных (RPO)**: Минимальная
- **Стоимость**: Высокая (постоянные ресурсы)
- **Примеры**: Database cluster с репликацией, active-active конфигурация

### Холодный резерв (Cold Standby)

- **Готовность**: Требуется время на развёртывание
- **Синхронизация**: Частичная / отсутствует
- **Время переключения (RTO)**: Высокое (минуты – часы)
- **Потеря данных (RPO)**: Возможна потеря данных
- **Стоимость**: Низкая (ресурсы неактивны)
- **Примеры**: Резервные серверы на другом дата-центре, восстановление из бэкапа

### Сценарий теста отказоустойчивости (Database Cluster)

**Цель**: Проверить автоматическое переключение при отказе главного узла

**Шаги**:

1. Убедиться, что данные синхронизируются на резервной базе
2. Запустить типовую нагрузку (200 человек одновременно)
3. Отключить главную базу от сети
4. Проверить переключение на резервную базу
5. Убедиться, что сайт снова работает, данные не потеряны

**Метрики**:

- RTO (Recovery Time Objective): Сколько сайт был недоступен (допустимо 30-60 сек)
- RPO (Recovery Point Objective): Сколько данных потеряно (должно быть 0)
- Latency: Не начал ли сайт тормозить после переключения
- Ошибки: Сколько пользователей получили ошибку 500

## DevTools и инструменты разработчика

### Основные функции

#### 1. Изменение текста через DevTools

- **Путь**: DevTools → Elements → редактировать HTML
- **Применение**: Проверка, как интерфейс ведет себя с разными текстами

#### 2. Coverage (Неиспользуемый код)

- **Путь**: DevTools → Coverage
- **Применение**: Найти неиспользуемые CSS и JS
- **Оптимизация**: Удалить лишний код, улучшить скорость загрузки

#### 3. Замеры скорости загрузки

- **Fast 4G**: 5 миллисекунд
- **Slow 4G**: 7 миллисекунд
- **300 kb/s**: 20 миллисекунд
- **Инструмент**: DevTools → Network → Throttling

#### 4. Блокировка ресурсов

- **Путь**: DevTools → Network → Right-click request → Block URL
- **Применение**: Проверить, как сайт работает без изображений, CSS или JS

## Кросс-браузерное тестирование

### Типичные различия между браузерами

| Элемент                                         | Отличие                                                       | Браузеры    | Причина                                                                      |
| ------------------------------------------------------ | -------------------------------------------------------------------- | ------------------- | ----------------------------------------------------------------------------------- |
| **Шапка сайта (border-radius)**        | Тёмный артефакт на скругленных углах | Safari на macOS   | Особенности рендеринга графики (композитинг) |
| **Сетка карточек (padding/margin)** | Разные отступы между карточками          | Firefox             | Движок Gecko иначе интерпретирует CSS                      |
| **Полоса прокрутки**              | Firefox более толстая и контрастная          | Chrome/Edge/Safari  | Разные стили для scrollbar по умолчанию                    |
| **Интерактивная карта**        | Safari более плавный скролл                        | Chrome/Firefox/Edge | WebKit лучше интегрирован с macOS                                 |

### Рекомендации

- ✅ Тестировать на Chrome, Firefox, Safari, Edge
- ✅ Проверять на разных ОС (Windows, macOS, Linux)
- ✅ Использовать BrowserStack для удалённого тестирования
- ✅ Проверять responsive design на разных разрешениях

## HTTPS/TLS Handshake

### Этапы установления соединения

#### 1. DNS-запрос

- Получить IP-адрес сервера (A или AAAA запись)
- **Риск**: DNS spoofing без защищённого DNS

#### 2. TCP 3-way Handshake

- SYN (Client → Server)
- SYN-ACK (Server → Client)
- ACK (Client → Server)
- **Риск**: TCP не защищает от перехвата

#### 3. ClientHello (начало TLS)

- Версия TLS
- Список cipher suites
- Случайное число клиента (client random)
- Расширения (SNI, ALPN)
- **Риск**: SNI видно в plaintext

#### 4. ServerHello + Сертификат

- Выбранная версия TLS
- Выбранная cipher suite
- Случайное число сервера (server random)
- X.509 сертификат
- ECDHE параметры

#### 5. Проверка сертификата клиентом

- Проверить подпись CA
- Проверить CN / SAN (соответствие домену)
- Проверить срок действия
- Проверить отзыв (OCSP / CRL)
- **Риск**: Без проверки возможна MITM атака

#### 6. Обмен ключами (ECDHE)

- Клиент и сервер обмениваются временными публичными ключами
- Вычисляют общий секрет
- Из секрета формируются симметричные сеансовые ключи
- **Преимущество**: Perfect Forward Secrecy

#### 7. Finished-сообщения

- Обе стороны отправляют зашифрованные Finished
- Содержат хеш всех предыдущих этапов
- **Цель**: Подтвердить целостность handshake

#### 8. Передача HTTP-данных

- Используются симметричные алгоритмы (AES-GCM, ChaCha20)
- Применяется AEAD для конфиденциальности и целостности
- **Защита**: Прослушивание, подмена данных, кража cookie/токенов

## Куки и флаги безопасности

### Флаги куки

#### 1. Secure

- **Назначение**: Кука передается только по HTTPS
- **Применение**: Защита от перехвата в открытых сетях (Wi-Fi)
- **Использование**: ✅ Для всех чувствительных кук

#### 2. HttpOnly

- **Назначение**: Кука недоступна для JavaScript на клиентской стороне
- **Применение**: Защита от XSS-атак
- **Использование**: ✅ Для кук с токенами авторизации

#### 3. SameSite

- **Strict**: Кука отправляется только на том же сайте (максимальная безопасность)
- **Lax**: Кука не отправляется при кросс-доменных запросах, но отправляется при переходах по ссылкам (баланс)
- **None**: Кука отправляется везде, но требует Secure флаг (для кросс-доменной авторизации)
- **Применение**: Защита от CSRF
- **Рекомендация**: ✅ Lax как умолчание, Strict для критичных данных

### Примеры из Ozon

```
__Secure-ab-group: Secure, SameSite=Lax
  → A/B-тесты, нужна для консистентности эксперимента

__Secure-access-token: Secure, SameSite=Lax, HttpOnly
  → Токен авторизации, максимальная защита

__Secure-ETC: HttpOnly, SameSite=None, Secure
  → Служебный идентификатор, передается на другие сайты

__Secure-refresh-token: HttpOnly, SameSite=Lax, Secure
  → Токен обновления, нужна для возобновления сессии при переходах
```

## Практические рекомендации

1. **Архитектура**: Планируйте отказоустойчивость с самого начала
2. **DevTools**: Используйте для отладки и оптимизации
3. **Кросс-браузерность**: Тестируйте на основных браузерах и ОС
4. **HTTPS**: Всегда используйте TLS 1.2/1.3
5. **Куки**: Правильно настраивайте флаги безопасности
6. **Мониторинг**: Отслеживайте RTO, RPO и другие метрики
