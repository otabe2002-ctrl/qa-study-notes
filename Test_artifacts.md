## Основные артефакты тестирования

**Артефакты** - это документы, которые создаются и используются во время тестирования. Они являются основой для отслеживания прогресса и обеспечения качества.

---

## Тест-кейсы (Test Cases)

### Структура тест-кейса

- **ID**: Уникальный идентификатор (TC_AUTH_POS-01)
- **Название**: Краткое описание того, что проверяется
- **Описание**: Расширенное объяснение цели теста
- **Предусловия**: Что должно быть перед выполнением теста
- **Окружение**: ОС, браузер, версия приложения, URL
- **Шаги выполнения**: Детальные действия
- **Ожидаемый результат**: Что должно произойти
- **Фактический результат**: Что произошло на самом деле
- **Статус**: Passed / Failed / Not Tested
- **Исполнитель**: Имя тестировщика
- **Комментарии**: Дополнительная информация

### Типы тест-кейсов

#### Позитивные тест-кейсы

- **Цель**: Проверка корректного поведения при валидных входных данных
- **Пример**: Успешная авторизация с правильными логином и паролем
- **Ожидаемый результат**: Система работает как предусмотрено

#### Негативные тест-кейсы

- **Цель**: Проверка поведения при некорректных входных данных или ошибочных сценариях
- **Примеры**:
  - Авторизация заблокированного пользователя
  - Авторизация с несуществующим логином/паролем
  - Авторизация с пустыми полями
  - Авторизация с лишними пробелами
- **Ожидаемый результат**: Система отклоняет запрос с понятным сообщением об ошибке

## Тест-план (Test Plan)

### Назначение

Документ, который описывает объем, подход, ресурсы и график тестирования проекта.

### Основные разделы

1. **Вступление**: Цель тестирования и контекст проекта
2. **Функциональности, которые будут протестированы**: Перечень компонентов и функций
3. **Функциональности, которые НЕ будут протестированы**: Исключения и причины
4. **Тестовые единицы**: Логические блоки для тестирования (Header, Footer, Формы)
5. **Подходы и техники**: Функциональное, UI, кросс-браузерное, адаптивное тестирование
6. **Критерии приёмки (Pass/Fail)**: Условия успеха и провала
7. **Приостановка/Возобновление**: Когда приостановить и как возобновить тестирование
8. **Задачи тестировщиков**: Обязанности и ответственность

### Пример из практики (School21)

- **Объект**: Раздел "Корпоративное обучение"
- **Подход**: Функциональное, UI, кросс-браузерное, адаптивное
- **Критерии Pass**: Страница загружается, форма отправляется, нет критических дефектов
- **Критерии Fail**: Страница не загружается, форма не отправляется, критические дефекты найдены

## Чек-листы (Checklists)

### Назначение

Упорядоченный список проверок, которые нужно выполнить. Проще и быстрее, чем тест-кейсы.

### Структура

```
☐ Проверка 1
☐ Проверка 2
☐ Проверка 3
```

### Примеры проверок

- ✅ Доступность сайта
- ✅ Корректность URL
- ✅ UTF-8 кодировка
- ✅ Отображение блоков
- ✅ Работа ссылок
- ✅ Функциональность форм
- ✅ Кроссбраузерность (Chrome, Firefox, Safari, Edge)
- ✅ Мобильная адаптивность
- ✅ Скорость загрузки

### Преимущества

- Быстрое выполнение
- Легко обновлять
- Хорошо для регрессионного тестирования

## Тестовые сценарии (Test Scenarios)

### Структура сценария

1. **Название**: Описание сценария
2. **Предусловия**: Начальное состояние
3. **Сценарий**: Последовательность действий
4. **Ожидаемый результат**: Финальное состояние
5. **Постусловия**: Состояние после завершения

### Пример из School21: Процесс поступления

**ТС-01. Оформление заявки на поступление**

- Пользователь имеет Sber ID и не зарегистрирован
- Вход через Sber ID → Регистрация на сайте
- Аккаунт создан успешно

**ТС-02. Прохождение вступительного этапа**

- Пользователь зарегистрирован
- Открывает страницу вступительного испытания → Проходит игру
- Получает возможность просмотреть видео

**ТС-03. Просмотр видео**

- Пользователь прошел игру полностью
- Переходит на вкладку с видео → Просматривает видео
- Получает возможность заполнить анкету

**ТС-04. Заполнение анкеты**

- Пользователь прошел игру и просмотрел видео
- Заполняет все обязательные поля → Отправляет анкету
- Анкета успешно отправлена, данные сохранены

## Тестовые данные

### Источники тестовых данных

1. **Online File Generator**: Генерация файлов нужного размера
2. **Mockaroo**: Генератор случайных данных (JSON/CSV/SQL)
3. **TestDataHub**: Генератор фейковых данных
4. **Ручное создание**: Для специфических сценариев

### Примеры тестовых данных

```
| id | first_name    | last_name     | email            | phone           |
|----|---------------|---------------|------------------|-----------------|
| 1  | Александр     | Попов         | 0r4@mail.ru      | +7 955 554 61 35|
| 2  | Александр     | Павлов        | uvy@gmail.com    | +7 928 846 40 54|
| 3  | Сергей        | Новиков       | en4@mail.ru      | +7 941 668 41 54|
```

### Правила для тестовых данных

- **Репрезентативность**: Должны отражать реальные данные пользователей
- **Достаточность**: Достаточно для покрытия всех сценариев
- **Безопасность**: Не содержать реальные личные данные
- **Документированность**: Хранить в версионируемом виде

## Лучшие практики

1. **Организация артефактов**

   - Структурировать по проектам и модулям
   - Использовать версионирование
   - Хранить в centralized system (TMS)
2. **Поддержка актуальности**

   - Обновлять при изменении требований
   - Удалять устаревшие тест-кейсы
   - Регулярно ревьюировать
3. **Координация с разработкой**

   - Вовлекать разработчиков в создание тест-кейсов
   - Согласовывать критерии приёмки
   - Обсуждать граничные случаи
4. **Метрики и отчетность**

   - Отслеживать процент покрытия требований
   - Подсчитывать количество дефектов по статусам
   - Анализировать эффективность тестирования
